# Comparative Analysis: `src-no-skills` vs `src-dotnet-artisan` vs `src-dotnet-webapi`

All three folders contain the same three ASP.NET Core Web API apps (FitnessStudioApi, LibraryApi, VetClinicApi), all targeting **.NET 10** with **EF Core + SQLite**. The `src-no-skills` apps were generated by Copilot **without** any custom skills. The `src-dotnet-artisan` apps were generated **with** the dotnet-artisan skill set (using-dotnet → dotnet-advisor → dotnet-csharp + dotnet-api). The `src-dotnet-webapi` apps were generated **with** a custom `dotnet-webapi` skill focused on ASP.NET Core Web API best practices.

This report describes the high-level differences in implementation quality and patterns across all three approaches.

---

## Executive Summary

The three approaches produce working applications with the same functionality but differ significantly in code quality, consistency, and adherence to modern .NET best practices. The `src-dotnet-webapi` skill produces the most **operationally robust** code (CancellationToken, AsNoTracking, rich OpenAPI). The `src-dotnet-artisan` skill produces the most **idiomatically modern** C# (sealed types, records, minimal API typed results). The `src-no-skills` version is **functional but dated**, missing several best practices consistently.

| Dimension | `src-no-skills` | `src-dotnet-artisan` | `src-dotnet-webapi` |
|-----------|-----------------|----------------------|---------------------|
| API style | Controllers (all 3) | Minimal APIs (all 3) | Minimal APIs (all 3) |
| Sealed types | ❌ Not used | ✅ All classes sealed | ⚠️ Only `ApiExceptionHandler` sealed |
| Primary constructors | ❌ Traditional constructors | ✅ C# 12 primary constructors | ✅ C# 12 primary constructors |
| DTO types | Mixed (records + classes) | Sealed records (immutable) | Mutable classes |
| Service interfaces | ✅ All 3 apps use interfaces | ❌ No interfaces (concrete DI) | ✅ All 3 apps use interfaces |
| CancellationToken | ❌ Not passed | ⚠️ Inconsistent (2 of 3 apps) | ✅ All 3 apps, full chain |
| AsNoTracking() | ❌ Not used | ⚠️ Inconsistent (1 of 3 apps) | ✅ All 3 apps, systematic |
| DTO naming | `*Dto` suffix for everything | `*Request` / `*Response` | `*Request` / `*Response` |
| Return types | `IReadOnlyList<T>` / `IEnumerable<T>` | `IReadOnlyList<T>` in pagination | `List<T>` throughout |
| Seeder pattern | Static methods (mixed sync/async) | Static async methods | Static async methods |
| Exception handling | Custom exception classes | IExceptionHandler + mixed strategies | IExceptionHandler + built-in exceptions |
| PaginatedResponse | Mixed (record + class) | Sealed record with IReadOnlyList | Mutable class with `Create()` factory |
| OpenAPI metadata | Moderate (XML comments + attributes) | Minimal (`.WithTags()` only) | ✅ Rich (`.WithName()`, `.WithSummary()`, `.WithDescription()`) |
| Collection init | `new List<T>()` | `[]` (C# 12) | `[]` (C# 12) |
| File organization | Controllers/, DTOs/, Services/ | Endpoints/, DTOs/, Services/ | Endpoints/, DTOs/, Services/ |

## 1. API Style: Controllers vs Minimal APIs

**no-skills**: All three apps use the traditional controller-based approach with `[ApiController]`, `ControllerBase`, and attribute routing.

```csharp
// src-no-skills/FitnessStudioApi/Controllers/MembersController.cs
[ApiController]
[Route("api/members")]
[Produces("application/json")]
public class MembersController : ControllerBase
{
    private readonly IMemberService _service;
    public MembersController(IMemberService service) => _service = service;

    [HttpGet]
    [ProducesResponseType<PagedResult<MemberDto>>(200)]
    public async Task<IActionResult> GetAll(
        [FromQuery] string? search, [FromQuery] bool? isActive,
        [FromQuery] int page = 1, [FromQuery] int pageSize = 10)
        => Ok(await _service.GetAllAsync(search, isActive, page, pageSize));
}
```

**dotnet-artisan**: All three apps use minimal APIs with static extension methods, `MapGroup()`, and typed results.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Endpoints/MemberEndpoints.cs
public static class MemberEndpoints
{
    public static RouteGroupBuilder MapMemberEndpoints(this IEndpointRouteBuilder routes)
    {
        var group = routes.MapGroup("/api/members").WithTags("Members");

        group.MapGet("/", async Task<Ok<PagedResponse<MemberResponse>>> (
            string? search, bool? isActive, int? page, int? pageSize,
            MemberService service, CancellationToken ct) =>
        {
            var p = page is null or < 1 ? 1 : page.Value;
            var ps = pageSize is null or < 1 or > 100 ? 20 : pageSize.Value;
            return TypedResults.Ok(await service.GetAllAsync(search, isActive, p, ps, ct));
        });
        return group;
    }
}
```

**dotnet-webapi**: All three apps use minimal APIs with the same `MapGroup()` pattern but with rich OpenAPI metadata chaining.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
public static class MemberEndpoints
{
    public static void MapMemberEndpoints(this WebApplication app)
    {
        var group = app.MapGroup("/api/members").WithTags("Members");

        group.MapGet("/", async (string? search, bool? isActive, int? page, int? pageSize,
            IMemberService service, CancellationToken ct) =>
        {
            var p = Math.Max(1, page ?? 1);
            var ps = Math.Clamp(pageSize ?? 20, 1, 100);
            return Results.Ok(await service.GetAllAsync(search, isActive, p, ps, ct));
        })
        .WithName("GetMembers")
        .WithSummary("List members")
        .WithDescription("Returns a paginated list of members with optional search and active filter.")
        .Produces<PaginatedResponse<MemberResponse>>(StatusCodes.Status200OK);
    }
}
```

**Notable**: The dotnet-artisan version uses `TypedResults` (strongly-typed return) with `Task<Ok<T>>` signatures, while dotnet-webapi uses `Results` (weakly-typed) with `.Produces<T>()` metadata. The dotnet-artisan approach embeds OpenAPI metadata in the return type itself; dotnet-webapi chains it explicitly.

**Verdict**: Both minimal API approaches are more modern than controllers. **dotnet-artisan** wins on type safety (TypedResults). **dotnet-webapi** wins on OpenAPI richness. **no-skills** is functional but uses the older pattern. Rank: dotnet-artisan ≈ dotnet-webapi > no-skills.

## 2. Sealed Types

**dotnet-artisan**: Every class — models, services, DTOs, middleware — is declared `sealed` across all three apps.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Models/Member.cs
public sealed class Member
{
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    // ...
    public ICollection<Membership> Memberships { get; set; } = [];
}
```

**dotnet-webapi**: Only the `ApiExceptionHandler` is sealed (marked `internal sealed`). All models, services, and DTOs are not sealed.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Models/Member.cs
public class Member
{
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    // ...
    public ICollection<Membership> Memberships { get; set; } = [];
}
```

**no-skills**: No classes are sealed anywhere. All use plain `public class`.

```csharp
// src-no-skills/FitnessStudioApi/Models/Member.cs
public class Member
{
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    // ...
    public ICollection<Membership> Memberships { get; set; } = new List<Membership>();
}
```

**Verdict**: Sealing by default is a .NET best practice (CA1852 analyzer rule), preventing unintended inheritance and enabling JIT devirtualization. **dotnet-artisan** is the only one that applies this consistently. Rank: dotnet-artisan > dotnet-webapi > no-skills.

## 3. Primary Constructors (C# 12)

**dotnet-artisan** and **dotnet-webapi**: Both consistently use primary constructors for dependency injection in services, reducing boilerplate.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs
public sealed class MemberService(StudioDbContext db)
{
    // db is captured directly — no field declarations needed
}

// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
public class MemberService(FitnessDbContext db, ILogger<MemberService> logger) : IMemberService
{
    // db and logger are captured directly
}
```

**no-skills**: Uses traditional constructor injection with explicit field declarations and arrow-bodied syntax.

```csharp
// src-no-skills/FitnessStudioApi/Services/MemberService.cs
public class MemberService : IMemberService
{
    private readonly FitnessDbContext _db;

    public MemberService(FitnessDbContext db) => _db = db;
}
```

**Verdict**: Primary constructors are the idiomatic C# 12+ pattern. Both skill-generated versions are modern; no-skills is dated. Rank: dotnet-artisan = dotnet-webapi > no-skills.

## 4. DTOs: Records vs Classes

**dotnet-artisan**: All DTOs across all three apps are `sealed record` types, giving immutability, value-based equality, and concise syntax. Request DTOs use `init` properties for validation attributes.

```csharp
// src-dotnet-artisan/FitnessStudioApi/DTOs/Dtos.cs
public sealed record MemberResponse(
    int Id, string FirstName, string LastName, string Email, string Phone,
    DateOnly DateOfBirth, string EmergencyContactName, string EmergencyContactPhone,
    DateOnly JoinDate, bool IsActive, MembershipSummary? ActiveMembership);

public sealed record CreateMemberRequest(
    [Required, MaxLength(100)] string FirstName,
    [Required, MaxLength(100)] string LastName,
    [Required, EmailAddress] string Email,
    [Required] string Phone,
    DateOnly DateOfBirth,
    [Required, MaxLength(200)] string EmergencyContactName,
    [Required] string EmergencyContactPhone);
```

**dotnet-webapi**: All DTOs are mutable classes with `{ get; set; }` properties. Not records, not sealed.

```csharp
// src-dotnet-webapi/FitnessStudioApi/DTOs/MemberDtos.cs
public class CreateMemberRequest
{
    [Required, MaxLength(100)]
    public string FirstName { get; set; } = string.Empty;
    [Required, MaxLength(100)]
    public string LastName { get; set; } = string.Empty;
    [Required, EmailAddress]
    public string Email { get; set; } = string.Empty;
    // ...
}

public class MemberResponse
{
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    // ...
}
```

**no-skills**: Mixed — FitnessStudioApi uses positional **records** with `*Dto` naming, while LibraryApi and VetClinicApi use **mutable classes**.

```csharp
// src-no-skills/FitnessStudioApi/DTOs/Dtos.cs — records
public record MemberDto(
    int Id, string FirstName, string LastName, string Email, string Phone,
    DateOnly DateOfBirth, string EmergencyContactName, string EmergencyContactPhone,
    DateOnly JoinDate, bool IsActive, DateTime CreatedAt, DateTime UpdatedAt);

// src-no-skills/VetClinicApi/DTOs/Dtos.cs — mutable classes
public class OwnerCreateDto
{
    public string FirstName { get; set; } = string.Empty;
    // ...
}
```

**Verdict**: Records are the recommended type for DTOs in modern C#. They enforce immutability and reduce boilerplate. **dotnet-artisan** is the strongest and most consistent. **no-skills** is inconsistent (records in one app, classes in two). **dotnet-webapi** consistently uses classes, which is less ideal but at least uniform. Rank: dotnet-artisan > no-skills (partial) > dotnet-webapi.

## 5. DTO Naming Convention

**dotnet-artisan** and **dotnet-webapi**: Both use a `Request`/`Response` naming convention that clearly distinguishes input from output.

| Role | Convention | Example |
|------|-----------|---------|
| Input | `Create{Entity}Request`, `Update{Entity}Request` | `CreateMemberRequest` |
| Output | `{Entity}Response` | `MemberResponse` |

**no-skills**: Uses `Dto` suffix for everything (FitnessStudio) or mixed `Create*Dto` / `*ResponseDto` (VetClinic).

| Role | Convention | Example |
|------|-----------|---------|
| Input | `Create{Entity}Dto`, `Update{Entity}Dto` | `CreateMemberDto`, `OwnerCreateDto` |
| Output | `{Entity}Dto`, `{Entity}ResponseDto` | `MemberDto`, `OwnerResponseDto` |

**Verdict**: The `Request`/`Response` convention is clearer and more widely adopted. Both skill versions use it consistently. Rank: dotnet-artisan = dotnet-webapi > no-skills.

## 6. Service Abstraction (Interfaces)

**no-skills** and **dotnet-webapi**: All three apps consistently use the interface + implementation pattern for services, enabling testability and adherence to the Dependency Inversion Principle.

```csharp
// src-no-skills/FitnessStudioApi/Program.cs
builder.Services.AddScoped<IMembershipPlanService, MembershipPlanService>();
builder.Services.AddScoped<IMemberService, MemberService>();

// src-dotnet-webapi/FitnessStudioApi/Program.cs
builder.Services.AddScoped<IMembershipPlanService, MembershipPlanService>();
builder.Services.AddScoped<IMemberService, MemberService>();
```

**dotnet-artisan**: None of the three apps use interfaces. Concrete services are injected directly.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Program.cs
builder.Services.AddScoped<MembershipPlanService>();
builder.Services.AddScoped<MemberService>();
builder.Services.AddScoped<MembershipService>();
```

**Verdict**: This is a debatable dimension. Interface-based DI is the industry standard for testability (mocking) and SOLID compliance. However, the dotnet-artisan approach of sealed concrete services is defensible for small apps — it reduces ceremony and the sealed keyword prevents subclassing anyway. For enterprise codebases, interfaces are preferred. Rank: no-skills = dotnet-webapi > dotnet-artisan (for testability).

## 7. CancellationToken Propagation

**dotnet-webapi**: `CancellationToken` is forwarded from endpoints through services to all EF Core async calls in **all three apps** — the most consistent of the three.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
public async Task<PaginatedResponse<MemberResponse>> GetAllAsync(
    string? search, bool? isActive, int page, int pageSize, CancellationToken ct)
{
    var query = db.Members.AsNoTracking().AsQueryable();
    // ...
    var totalCount = await query.CountAsync(ct);
    var items = await query
        .OrderBy(m => m.LastName).ThenBy(m => m.FirstName)
        .Skip((page - 1) * pageSize).Take(pageSize)
        .Select(m => MapToResponse(m))
        .ToListAsync(ct);
    return PaginatedResponse<MemberResponse>.Create(items, page, pageSize, totalCount);
}
```

**dotnet-artisan**: CancellationToken is used in FitnessStudioApi and VetClinicApi, but **not in LibraryApi** — an inconsistency.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs — ✅ uses ct
public async Task<MemberResponse?> GetByIdAsync(int id, CancellationToken ct = default)
{
    var member = await db.Members.FindAsync([id], ct);
    // ...
}

// src-dotnet-artisan/LibraryApi/Services/LibraryService.cs — ❌ no ct parameter
public async Task<PaginatedResponse<AuthorResponse>> GetAuthorsAsync(
    string? search, int page, int pageSize)
{
    var totalCount = await query.CountAsync();
    var items = await query.Skip((page - 1) * pageSize).Take(pageSize)
        .Select(a => new AuthorResponse(...)).ToListAsync();
}
```

**no-skills**: CancellationToken is **not passed at all** in any of the three apps.

```csharp
// src-no-skills/FitnessStudioApi/Services/MemberService.cs
public async Task<MemberDetailDto?> GetByIdAsync(int id)
{
    var member = await _db.Members.FindAsync(id);
    // No CancellationToken anywhere
}
```

**Verdict**: CancellationToken forwarding is critical for graceful request cancellation and resource cleanup. The no-skills version will continue processing abandoned requests. **dotnet-webapi** is the most consistent; **dotnet-artisan** has a gap in LibraryApi. Rank: dotnet-webapi > dotnet-artisan > no-skills.

## 8. AsNoTracking() for Read Queries

**dotnet-webapi**: Consistently applies `.AsNoTracking()` on all read-only queries across **all three apps**.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
var query = db.Members.AsNoTracking().AsQueryable();

// src-dotnet-webapi/LibraryApi/Services/BookService.cs
var query = db.Books.AsNoTracking()
    .Include(b => b.BookAuthors).ThenInclude(ba => ba.Author)
    .Include(b => b.BookCategories).ThenInclude(bc => bc.Category)
    .AsQueryable();
```

**dotnet-artisan**: Only VetClinicApi uses AsNoTracking. FitnessStudioApi and LibraryApi do **not**.

```csharp
// src-dotnet-artisan/VetClinicApi/Services/PetService.cs — ✅ uses AsNoTracking
var query = db.Pets.AsNoTracking().AsQueryable();

// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs — ❌ no AsNoTracking
var query = db.Members.AsQueryable();
```

**no-skills**: Does not use `AsNoTracking()` on any read queries in any of the three apps.

**Verdict**: `AsNoTracking()` is a well-known EF Core performance optimization for reads, avoiding unnecessary change-tracking overhead. **dotnet-webapi** is the only version that applies this systematically. Rank: dotnet-webapi > dotnet-artisan > no-skills.

## 10. Exception Handling Strategy

This is where the three approaches diverge most significantly.

**no-skills**: Uses **custom exception classes** (`BusinessRuleException`, `ConflictException`, `NotFoundException`) with convention-based middleware in FitnessStudio and VetClinic, and `IExceptionHandler` in LibraryApi. VetClinic also uses result tuples.

```csharp
// src-no-skills/FitnessStudioApi/Middleware/GlobalExceptionMiddleware.cs
public class GlobalExceptionMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<GlobalExceptionMiddleware> _logger;

    public GlobalExceptionMiddleware(RequestDelegate next, ILogger<GlobalExceptionMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        try { await _next(context); }
        catch (Exception ex) { await HandleExceptionAsync(context, ex); }
    }

    private async Task HandleExceptionAsync(HttpContext context, Exception exception)
    {
        var statusCode = exception switch
        {
            BusinessRuleException => StatusCodes.Status400BadRequest,
            ConflictException => StatusCodes.Status409Conflict,
            NotFoundException => StatusCodes.Status404NotFound,
            _ => StatusCodes.Status500InternalServerError
        };
        // ... writes ProblemDetails
    }
}
```

**dotnet-artisan**: Uses `IExceptionHandler` (.NET 8+) across all apps. Exception strategies vary — FitnessStudio uses `InvalidOperationException` with error tuples in endpoints; VetClinic uses built-in exceptions.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Middleware/GlobalExceptionHandler.cs
public sealed class GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken cancellationToken)
    {
        logger.LogError(exception, "Unhandled exception: {Message}", exception.Message);
        var problemDetails = new ProblemDetails
        {
            Status = (int)HttpStatusCode.InternalServerError,
            Title = "An unexpected error occurred",
            Detail = httpContext.RequestServices.GetRequiredService<IWebHostEnvironment>().IsDevelopment()
                ? exception.Message
                : "An internal error occurred. Please try again later.",
            Instance = httpContext.Request.Path
        };
        httpContext.Response.StatusCode = problemDetails.Status.Value;
        await httpContext.Response.WriteAsJsonAsync(problemDetails, cancellationToken);
        return true;
    }
}
```

**dotnet-webapi**: Uses `IExceptionHandler` with **pattern-matching switch on built-in .NET exceptions** — no custom exception classes needed. Identical pattern in all three apps.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Middleware/ApiExceptionHandler.cs
internal sealed class ApiExceptionHandler(ILogger<ApiExceptionHandler> logger) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken cancellationToken)
    {
        var (statusCode, title) = exception switch
        {
            KeyNotFoundException => (StatusCodes.Status404NotFound, "Not Found"),
            ArgumentException => (StatusCodes.Status400BadRequest, "Bad Request"),
            InvalidOperationException => (StatusCodes.Status409Conflict, "Conflict"),
            _ => (0, (string?)null)
        };

        if (statusCode == 0) return false;

        logger.LogWarning(exception, "Handled API exception: {Title}", title);
        httpContext.Response.StatusCode = statusCode;
        await httpContext.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = statusCode,
            Title = title,
            Detail = exception.Message,
            Instance = httpContext.Request.Path
        }, cancellationToken);
        return true;
    }
}
```

**Verdict**: The **dotnet-webapi** approach is the cleanest — it uses built-in .NET exception types (no custom classes), `IExceptionHandler` (modern pattern), and pattern matching. The `return false` for unhandled exceptions is elegant. **dotnet-artisan** uses the modern interface but catches everything as 500. **no-skills** is the most varied. Rank: dotnet-webapi > dotnet-artisan > no-skills.

## 11. Pagination Type Design

Each approach handles pagination differently:

**no-skills**: Mixed across apps — FitnessStudio uses a positional record with `IReadOnlyList<T>`, LibraryApi uses a mutable class with `List<T>`, VetClinic uses a mutable class with `IEnumerable<T>` and `HasPrevious`/`HasNext` properties.

```csharp
// src-no-skills/FitnessStudioApi/DTOs/Dtos.cs
public record PagedResult<T>(IReadOnlyList<T> Items, int TotalCount, int Page, int PageSize, int TotalPages);

// src-no-skills/VetClinicApi/DTOs/Dtos.cs
public class PagedResult<T>
{
    public IEnumerable<T> Items { get; set; } = Enumerable.Empty<T>();
    public int TotalCount { get; set; }
    public int TotalPages { get; set; }
    public int CurrentPage { get; set; }
    public int PageSize { get; set; }
    public bool HasPrevious => CurrentPage > 1;
    public bool HasNext => CurrentPage < TotalPages;
}
```

**dotnet-artisan**: Sealed records across all three apps, with `IReadOnlyList<T>` for items. VetClinic adds `HasNextPage`/`HasPreviousPage`.

```csharp
// src-dotnet-artisan/FitnessStudioApi/DTOs/Dtos.cs
public sealed record PagedResponse<T>(
    IReadOnlyList<T> Items, int Page, int PageSize, int TotalCount, int TotalPages);

// src-dotnet-artisan/VetClinicApi/DTOs/PagedResponse.cs
public sealed record PagedResponse<T>(
    List<T> Items, int Page, int PageSize, int TotalCount, int TotalPages,
    bool HasNextPage, bool HasPreviousPage);
```

**dotnet-webapi**: Mutable class with `List<T>`, `HasNextPage`/`HasPreviousPage`, and a static `Create()` factory method.

```csharp
// src-dotnet-webapi/FitnessStudioApi/DTOs/PaginatedResponse.cs
public class PaginatedResponse<T>
{
    public List<T> Items { get; set; } = [];
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalCount { get; set; }
    public int TotalPages { get; set; }
    public bool HasNextPage { get; set; }
    public bool HasPreviousPage { get; set; }

    public static PaginatedResponse<T> Create(List<T> items, int page, int pageSize, int totalCount)
    {
        var totalPages = (int)Math.Ceiling(totalCount / (double)pageSize);
        return new PaginatedResponse<T>
        {
            Items = items, Page = page, PageSize = pageSize,
            TotalCount = totalCount, TotalPages = totalPages,
            HasNextPage = page < totalPages,
            HasPreviousPage = page > 1
        };
    }
}
```

**Verdict**: The **dotnet-artisan** sealed record approach is the most type-safe (IReadOnlyList prevents mutation, sealed record prevents inheritance). The **dotnet-webapi** `Create()` factory is a nice pattern that encapsulates calculation. The **no-skills** version is the most inconsistent across apps. Rank: dotnet-artisan > dotnet-webapi > no-skills.

## 12. OpenAPI / Swagger Documentation

**dotnet-webapi**: The richest OpenAPI metadata — all endpoints use `.WithName()`, `.WithSummary()`, `.WithDescription()`, and `.Produces<T>()` with multiple status codes.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
group.MapPost("/", async (CreateMemberRequest request, IMemberService service, CancellationToken ct) =>
{
    var member = await service.CreateAsync(request, ct);
    return Results.Created($"/api/members/{member.Id}", member);
})
.WithName("CreateMember")
.WithSummary("Register a new member")
.WithDescription("Registers a new member. Must be at least 16 years old. Email must be unique.")
.Produces<MemberResponse>(StatusCodes.Status201Created)
.Produces(StatusCodes.Status400BadRequest)
.Produces(StatusCodes.Status409Conflict);
```

**no-skills**: Moderate — uses XML summary comments and `[ProducesResponseType]` attributes. LibraryApi has the richest setup with `SwaggerDoc` title/description.

```csharp
// src-no-skills/FitnessStudioApi/Controllers/MembersController.cs
/// <summary>Register a new member</summary>
[HttpPost]
```

## Summary: Impact of Skills

The three skill configurations (`none`, `dotnet-artisan`, `dotnet-webapi`) each produce **functional, architecturally sound code** but with meaningfully different quality profiles:

### `src-dotnet-webapi` — Best for Production Readiness

The `dotnet-webapi` custom skill produces the most **operationally robust** code. Its standout strengths are:

1. **CancellationToken forwarding** (all 3 apps, full chain) — graceful request cancellation
2. **AsNoTracking()** (all 3 apps, systematic) — EF Core performance optimization
3. **Rich OpenAPI metadata** (`.WithName()`, `.WithSummary()`, `.WithDescription()`) — API discoverability
4. **Built-in exception mapping** (no custom exception classes) — clean, maintainable error handling
5. **Interface-based DI** (all 3 apps) — testability and SOLID compliance

Its weakness: mutable DTOs (`class` instead of `record`) and no `sealed` on most types.

### `src-dotnet-artisan` — Best for Modern C# Idioms

The `dotnet-artisan` skill set produces the most **idiomatically modern** C# code. Its standout strengths are:

1. **Sealed types everywhere** — JIT optimization and design intent
2. **Sealed records for DTOs** — immutability, value equality, concise syntax
3. **TypedResults in minimal APIs** — compile-time OpenAPI inference
4. **Collection expressions** (`[]`) — modern C# 12 syntax
5. **Primary constructors** — reduced boilerplate (shared with dotnet-webapi)

Its weakness: inconsistent CancellationToken (2 of 3 apps) and AsNoTracking (1 of 3 apps), no service interfaces.

### `src-no-skills` — Functional but Dated

The no-skills version produces working code but misses ~15 specific best practices:

1. **No CancellationToken** — abandoned requests continue processing
2. **No AsNoTracking()** — unnecessary change-tracking overhead on reads
3. **No sealed types** — missed JIT optimization opportunity
4. **No primary constructors** — verbose boilerplate
5. **Controller-based APIs** — functional but older pattern

Its strength: consistent interface usage across all apps and reasonable architecture.

### Overall Ranking by Dimension

| Dimension | Winner | Runner-up | Third |
|-----------|--------|-----------|-------|
| CancellationToken | dotnet-webapi | dotnet-artisan | no-skills |
| AsNoTracking | dotnet-webapi | dotnet-artisan | no-skills |
| Sealed types | dotnet-artisan | dotnet-webapi | no-skills |
| DTO immutability | dotnet-artisan | no-skills | dotnet-webapi |
| Service interfaces | no-skills = dotnet-webapi | — | dotnet-artisan |
| OpenAPI richness | dotnet-webapi | no-skills | dotnet-artisan |
| Exception handling | dotnet-webapi | dotnet-artisan | no-skills |
| Primary constructors | dotnet-artisan = dotnet-webapi | — | no-skills |
| Pagination design | dotnet-artisan | dotnet-webapi | no-skills |

The ideal codebase would combine dotnet-webapi's operational practices (CancellationToken, AsNoTracking, OpenAPI) with dotnet-artisan's type design (sealed types, records, TypedResults). Neither skill alone produces perfect output, but both significantly outperform the no-skills baseline.
