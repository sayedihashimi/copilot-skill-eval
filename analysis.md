# Comparative Analysis: `src-no-skills` vs `src-dotnet-artisan`

Both folders contain the same three ASP.NET Core Web API apps (FitnessStudioApi, LibraryApi, VetClinicApi), all targeting **.NET 10** with **EF Core + SQLite**. The `src-no-skills` apps were generated by Copilot **without** any custom skills, while the `src-dotnet-artisan` apps were generated **with** the dotnet-artisan skill set (using-dotnet → dotnet-advisor → dotnet-csharp + dotnet-api).

This report describes the high-level differences in implementation quality and patterns.

---

## Executive Summary

The dotnet-artisan skill set produces code that is **more consistent, more idiomatic, and more aligned with modern C# and ASP.NET Core best practices**. The differences are not in overall architecture (both use a layered Controllers → Services → Data approach) but in the **quality of the code within that architecture**. The artisan version systematically applies ~15 specific practices that the no-skills version misses or applies inconsistently.

| Dimension | `src-no-skills` | `src-dotnet-artisan` |
|-----------|-----------------|----------------------|
| Sealed types | ❌ Not used | ✅ All classes sealed |
| Primary constructors | ❌ Traditional constructors | ✅ C# 12 primary constructors |
| DTO types | Classes (mutable) | Sealed records (immutable) |
| Service interfaces | Inconsistent (1 of 3 apps) | ✅ Always interface + implementation |
| CancellationToken | ❌ Not passed | ✅ Forwarded through all layers |
| AsNoTracking() | ❌ Not used | ✅ Used on all read queries |
| Return types | `List<T>` | `IReadOnlyList<T>` |
| DTO naming | `*Dto` suffix for everything | `*Request` / `*Response` distinction |
| Seeder pattern | Static methods | Injectable with ILogger |
| Middleware style | Convention-based | IMiddleware interface |
| PaginatedResponse | Class | Sealed record |
| Exception design | Custom exception classes | Varies (custom + built-in) |

---

## 1. Sealed Types

**dotnet-artisan**: Every class — models, services, controllers, middleware, DTOs — is declared `sealed`. This prevents unintended inheritance, enables JIT devirtualization optimizations, and is the recommended default per .NET design guidelines.

```csharp
// dotnet-artisan
public sealed class Member { ... }
public sealed class MemberService(...) : IMemberService { ... }
public sealed class MembersController(...) : ControllerBase { ... }
```

**no-skills**: No classes are sealed. All use plain `public class`.

```csharp
// no-skills
public class Member { ... }
public class MemberService { ... }
public class MembersController : ControllerBase { ... }
```

**Verdict**: dotnet-artisan is correct. Sealing by default is a .NET best practice (CA1852 analyzer rule).

---

## 2. Primary Constructors (C# 12)

**dotnet-artisan**: Consistently uses primary constructors for dependency injection in services, controllers, and middleware, reducing boilerplate.

```csharp
// dotnet-artisan
public sealed class MemberService(FitnessDbContext db, ILogger<MemberService> logger) : IMemberService
{
    // db and logger are captured directly — no field declarations needed
}
```

**no-skills**: Uses traditional constructor injection with explicit field declarations.

```csharp
// no-skills
public class MemberService
{
    private readonly FitnessDbContext _db;
    private readonly ILogger<MemberService> _logger;

    public MemberService(FitnessDbContext db, ILogger<MemberService> logger)
    {
        _db = db;
        _logger = logger;
    }
}
```

**Verdict**: Primary constructors are the idiomatic C# 12+ pattern. dotnet-artisan is more modern and concise.

---

## 3. DTOs: Records vs Classes

**dotnet-artisan**: All DTOs are `sealed record` types, giving immutability, value-based equality, and concise positional syntax.

```csharp
// dotnet-artisan
public sealed record MemberResponse(
    int Id, string FirstName, string LastName, string Email, ...);

public sealed record CreateMemberRequest
{
    [Required, MaxLength(100)]
    public string FirstName { get; init; } = string.Empty;
    ...
}
```

**no-skills**: All DTOs are plain mutable classes.

```csharp
// no-skills
public class MemberDto
{
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    ...
}
```

**Verdict**: Records are the recommended type for DTOs in modern C#. They enforce immutability and reduce boilerplate. dotnet-artisan is the stronger pattern.

---

## 4. Service Abstraction (Interfaces)

**dotnet-artisan**: All three apps consistently use the interface + implementation pattern for services. Interfaces are registered in DI.

```csharp
// dotnet-artisan (all 3 apps)
builder.Services.AddScoped<IMemberService, MemberService>();
```

**no-skills**: Only 1 of 3 apps (VetClinicApi) uses interfaces. The other two (FitnessStudioApi, LibraryApi) register concrete service types directly.

```csharp
// no-skills (FitnessStudio, Library)
builder.Services.AddScoped<MemberService>();
```

**Verdict**: Interface-based registration is a best practice for testability (mocking) and adherence to the Dependency Inversion Principle. dotnet-artisan is consistent; no-skills is not.

---

## 5. CancellationToken Propagation

**dotnet-artisan**: `CancellationToken` is forwarded from controllers through services to all EF Core async calls (`.ToListAsync(ct)`, `.FirstOrDefaultAsync(ct)`, `.SaveChangesAsync(ct)`).

```csharp
// dotnet-artisan
public async Task<MemberResponse?> GetByIdAsync(int id, CancellationToken ct)
{
    var member = await _db.Members.AsNoTracking().FirstOrDefaultAsync(m => m.Id == id, ct);
    ...
}
```

**no-skills**: CancellationToken is **not passed at all** — not in controller signatures, not in service methods, not in EF Core calls.

```csharp
// no-skills
public async Task<MemberDto?> GetByIdAsync(int id)
{
    var member = await _db.Members.FirstOrDefaultAsync(m => m.Id == id);
    ...
}
```

**Verdict**: CancellationToken forwarding is critical for graceful request cancellation and resource cleanup. The no-skills version will continue processing abandoned requests. This is a significant quality gap.

---

## 6. AsNoTracking() for Read Queries

**dotnet-artisan**: Consistently applies `.AsNoTracking()` on all read-only queries, avoiding unnecessary change-tracking overhead.

```csharp
// dotnet-artisan
var query = _db.Members.AsNoTracking().AsQueryable();
```

**no-skills**: Does not use `AsNoTracking()` on any read queries. All queries go through the change tracker.

**Verdict**: `AsNoTracking()` is a well-known EF Core performance optimization for reads. Its absence in no-skills is a missed best practice.

---

## 7. Return Type Precision (IReadOnlyList vs List)

**dotnet-artisan**: Returns `IReadOnlyList<T>` from service methods, signaling immutability to callers.

```csharp
// dotnet-artisan
Task<IReadOnlyList<BookingResponse>> GetUpcomingBookingsAsync(int memberId, CancellationToken ct);
```

**no-skills**: Returns `List<T>`, exposing the mutable concrete type.

```csharp
// no-skills
public async Task<List<BookingDto>> GetUpcomingBookingsAsync(int memberId)
```

**Verdict**: Returning `IReadOnlyList<T>` is more defensive and communicates API intent better. dotnet-artisan follows the guideline of exposing the least-powerful interface.

---

## 8. DTO Naming Convention

**dotnet-artisan**: Uses a `Request`/`Response` naming convention that clearly distinguishes input from output.

| Role | Convention | Example |
|------|-----------|---------|
| Input | `Create{Entity}Request`, `Update{Entity}Request` | `CreateMemberRequest` |
| Output | `{Entity}Response` or `{Entity}Dto` | `MemberResponse` |

**no-skills**: Uses `Dto` suffix for everything — inputs and outputs alike.

| Role | Convention | Example |
|------|-----------|---------|
| Input | `Create{Entity}Dto`, `Update{Entity}Dto` | `CreateMemberDto` |
| Output | `{Entity}Dto` | `MemberDto` |

**Verdict**: The `Request`/`Response` convention is clearer and more widely adopted in enterprise codebases. Both are valid, but the artisan convention communicates direction of data flow.

---

## 9. Data Seeder Design

**dotnet-artisan**: Seeder is an injectable service with `ILogger`, registered as transient, and uses async methods with `CancellationToken`.

```csharp
// dotnet-artisan
public sealed class DataSeeder(FitnessDbContext db, ILogger<DataSeeder> logger)
{
    public async Task SeedAsync(CancellationToken ct = default) { ... }
}
// Program.cs
builder.Services.AddTransient<DataSeeder>();
```

**no-skills**: Seeder uses a static `Initialize(DbContext)` method, with no logging or cancellation support.

```csharp
// no-skills
public static class SeedData
{
    public static void Initialize(LibraryDbContext context) { ... }
}
```

**Verdict**: The injectable pattern is more testable, supports structured logging, and follows DI conventions. The dotnet-artisan seeder also tends to produce richer seed data (e.g., VetClinic artisan seeder is 379 lines vs 99 lines in no-skills).

---

## 10. Middleware Implementation

**dotnet-artisan (LibraryApi)**: Uses the `IMiddleware` interface pattern — a typed, DI-friendly approach.

```csharp
// dotnet-artisan
public sealed class GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger) : IMiddleware
{
    public async Task InvokeAsync(HttpContext context, RequestDelegate next) { ... }
}
```

**no-skills**: Uses convention-based middleware with manual `RequestDelegate` injection.

```csharp
// no-skills
public class GlobalExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<GlobalExceptionHandlerMiddleware> _logger;
    public GlobalExceptionHandlerMiddleware(RequestDelegate next, ILogger<...> logger) { ... }
    public async Task InvokeAsync(HttpContext context) { ... }
}
```

**Verdict**: Both are valid ASP.NET Core patterns. The `IMiddleware` approach is cleaner with DI (avoids constructor injection of RequestDelegate), while convention-based is the original pattern. The artisan version tends to be shorter and more focused.

## 11. Exception Handling Strategy

This is where the two approaches diverge most **across apps** (inconsistent within each set):

| App | no-skills | dotnet-artisan |
|-----|-----------|---------------|
| FitnessStudio | Custom `BusinessRuleException(msg, statusCode)` | Custom `BusinessRuleException` + `ConflictException` with title |
| Library | Custom `BusinessRuleException` + `NotFoundException` | **Tuple returns** `(Entity?, Error?)` — no exceptions for business rules |
| VetClinic | Custom `BusinessRuleException` + `NotFoundException` | Uses **built-in** `ArgumentException`, `InvalidOperationException`, `KeyNotFoundException` |

**Notable**: The dotnet-artisan Library API uses a tuple-return pattern in services instead of throwing exceptions for business rule violations. This means controllers check error strings and return appropriate HTTP responses, rather than relying on middleware to translate exceptions. This is a more functional approach.

**Verdict**: The dotnet-artisan set is slightly more varied in its exception strategy across apps, but each individual approach is reasonable. The no-skills set is more uniform but always relies on custom exception classes.

```csharp
public ICollection<Pet> Pets { get; set; } = [];
```

**no-skills**: Also uses `[]` in some cases, but this is less consistent.

## 14. PaginatedResponse Type

**dotnet-artisan**: `sealed record` with `IReadOnlyList<T>` items and computed properties.

```csharp
public sealed record PaginatedResponse<T>(
    IReadOnlyList<T> Items, int Page, int PageSize, int TotalCount)
{
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasNextPage => Page < TotalPages;
    public bool HasPreviousPage => Page > 1;
}
```

**no-skills**: Mutable class with `IEnumerable<T>` items and settable properties.

```csharp
public class PaginatedResponse<T>
{
    public IEnumerable<T> Data { get; set; } = Enumerable.Empty<T>();
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalCount { get; set; }
    public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
}
```

**Verdict**: The artisan version is more type-safe (IReadOnlyList prevents lazy evaluation issues, sealed record prevents mutation).

## 15. What Both Versions Get Right

Despite the differences, both versions share strong foundations:

- ✅ Layered architecture (Controllers → Services → Data)
- ✅ EF Core with Fluent API configuration
- ✅ Unique indexes on key fields (email, ISBN, etc.)
- ✅ Enum-to-string conversion in EF Core
- ✅ RFC 7807 ProblemDetails error responses
- ✅ Data annotations for validation
- ✅ Swagger/OpenAPI integration
- ✅ Pagination for list endpoints
- ✅ Soft-delete patterns where appropriate
- ✅ DateOnly for date-only fields
- ✅ Comprehensive .http test files
- ✅ Realistic seed data
- ✅ JSON enum serialization as strings

---

## Summary: Impact of Skills

The dotnet-artisan skills (`using-dotnet` → `dotnet-advisor` → `dotnet-csharp` + `dotnet-api`) enforced a consistent set of **~15 coding standards** that the no-skills version missed or applied inconsistently. The most impactful differences are:

1. **CancellationToken forwarding** — Functional correctness for request cancellation
2. **AsNoTracking()** — Performance optimization for read queries
3. **Service interfaces** — Testability and adherence to SOLID principles
4. **Sealed types** — JIT optimization and design intent
5. **Immutable DTOs (records)** — Type safety and API clarity

The no-skills version produces **functional, architecturally sound code** — but the artisan version produces code that would pass a stricter code review with fewer comments.
