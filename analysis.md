# Comparative Analysis: src-no-skills, src-dotnet-webapi, src-dotnet-artisan, src-managedcode-dotnet-skills

## Introduction

This report compares four sets of ASP.NET Core Web API applications generated by GitHub Copilot under different skill configurations. Each configuration produced the same three apps — **FitnessStudioApi**, **LibraryApi**, and **VetClinicApi** — all targeting **.NET 10** with **EF Core + SQLite**. Each app was generated in an isolated agent context with no shared knowledge.

| Directory | Copilot Configuration |
|---|---|
| `src-no-skills` | No skills — baseline generation from prompt alone |
| `src-dotnet-webapi` | `dotnet-webapi` skill |
| `src-dotnet-artisan` | `dotnet-artisan` skills: `using-dotnet`, `dotnet-advisor`, `dotnet-csharp`, `dotnet-api`, `dotnet-tooling` |
| `src-managedcode-dotnet-skills` | `managedcode` skills: `dotnet-project-setup`, `dotnet-aspnet-core`, `dotnet-entity-framework-core`, `dotnet-microsoft-extensions`, `dotnet-minimal-apis`, `dotnet-modern-csharp` |

## 1. API Style — Controllers vs Minimal APIs

### What each directory does

**src-no-skills** — All three apps use traditional **controller-based** architecture:

```csharp
// src-no-skills/FitnessStudioApi/Controllers/MembersController.cs
[ApiController]
[Route("api/members")]
[Produces("application/json")]
public class MembersController : ControllerBase
{
    public MembersController(IMemberService service) { _service = service; }
}
```

**src-dotnet-webapi** — All three apps use **Minimal APIs** with `MapGroup` and extension methods:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
public static void MapMemberEndpoints(this IEndpointRouteBuilder app)
{
    var group = app.MapGroup("/api/members").WithTags("Members");
    group.MapGet("/", async Task<Ok<PaginatedResponse<MemberResponse>>> (
        IMemberService service, string? search = null,
        int page = 1, int pageSize = 20,
        CancellationToken ct = default) => { ... })
    .WithName("GetMembers")
    .WithSummary("List members");
}
```

**src-dotnet-artisan** — All three apps use **controllers**:

```csharp
// src-dotnet-artisan/FitnessStudioApi/Controllers/MembersController.cs
[ApiController]
[Route("api/members")]
[Produces("application/json")]
public class MembersController(IMemberService service) : ControllerBase
{
    [HttpGet]
    [ProducesResponseType<PaginatedResponse<MemberResponse>>(200)]
    public async Task<IActionResult> GetAll(...) => Ok(await service.GetAllAsync(...));
}
```

**src-managedcode-dotnet-skills** — **Mixed**: FitnessStudioApi and VetClinicApi use Minimal APIs; LibraryApi uses controllers:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
public static RouteGroupBuilder MapMemberEndpoints(this IEndpointRouteBuilder app)
{
    var group = app.MapGroup("/api/members").WithTags("Members");
    group.MapGet("/", async (string? search, bool? isActive, int page, int pageSize,
        IMemberService service, CancellationToken ct) => { ... })
    .WithName("GetMembers")
    .WithSummary("List members with search, filter, and pagination");
}
```

### Verdict

**`src-dotnet-webapi` is the strongest here** — consistent Minimal API usage across all apps aligns with Microsoft's recommended approach for new .NET 8+ projects. Minimal APIs offer better performance (no MVC middleware overhead), native `TypedResults` for OpenAPI inference, and a more concise endpoint definition syntax. `src-managedcode-dotnet-skills` partially adopts this but inconsistently. `src-no-skills` and `src-dotnet-artisan` default to the older controller pattern.

## 2. Sealed Types

### What each directory does

**src-no-skills** — Zero `sealed` usage anywhere. All models, services, and controllers use plain `class`.

**src-dotnet-webapi** — **Comprehensive** `sealed` usage across all three apps — models, services, DTOs, middleware:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Models/Member.cs
public sealed class Member { ... }

// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
public sealed class MemberService : IMemberService { ... }

// src-dotnet-webapi/FitnessStudioApi/Middleware/ApiExceptionHandler.cs
public sealed class ApiExceptionHandler : IExceptionHandler { ... }
```

**src-dotnet-artisan** — Only VetClinicApi uses `sealed` (models, services, controllers, DTOs, middleware). FitnessStudioApi and LibraryApi do not:

```csharp
// src-dotnet-artisan/VetClinicApi/Services/OwnerService.cs
public sealed class OwnerService(VetClinicDbContext db) : IOwnerService { ... }

// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs (NO sealed)
public class MemberService(FitnessDbContext db, ILogger<MemberService> logger) : IMemberService { ... }
```

**src-managedcode-dotnet-skills** — FitnessStudioApi and VetClinicApi use `sealed` extensively. LibraryApi does not:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/FitnessStudioApi/Services/MemberService.cs
public sealed class MemberService(FitnessDbContext db, ILogger<MemberService> logger) : IMemberService { ... }

// src-managedcode-dotnet-skills/LibraryApi/Services/BookService.cs (NO sealed)
public class BookService(LibraryDbContext context, ILogger<BookService> logger) : IBookService { ... }
```

### Verdict

**`src-dotnet-webapi` is the strongest** — marking classes `sealed` is a .NET performance best practice. The JIT can devirtualize calls on sealed types, improving call dispatch. It also signals design intent ("this class is not meant to be inherited"). The `dotnet-webapi` skill applies this uniformly, while other configurations are inconsistent.

## 3. Primary Constructors

### What each directory does

**src-no-skills** — **No primary constructors**. Uses traditional field assignment:

```csharp
// src-no-skills/FitnessStudioApi/Services/MemberService.cs
public class MemberService : IMemberService
{
    private readonly FitnessDbContext _context;
    private readonly ILogger<MemberService> _logger;

    public MemberService(FitnessDbContext context, ILogger<MemberService> logger)
    {
        _context = context;
        _logger = logger;
    }
}
```

**src-dotnet-webapi** — **Partial adoption**: FitnessStudioApi uses traditional constructors; LibraryApi and VetClinicApi use primary constructors:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs (traditional)
public sealed class MemberService : IMemberService
{
    private readonly FitnessDbContext _db;
    public MemberService(FitnessDbContext db) => _db = db;
}

// src-dotnet-webapi/LibraryApi/Services/BookService.cs (primary constructor)
public sealed class BookService(LibraryDbContext db, ILogger<BookService> logger) : IBookService
{
    // Uses db and logger directly — no backing fields
}
```

**src-dotnet-artisan** — **All three apps** use primary constructors:

```csharp
// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs
public class MemberService(FitnessDbContext db, ILogger<MemberService> logger) : IMemberService
{
    public async Task<PaginatedResponse<MemberResponse>> GetAllAsync(...)
    {
        var query = db.Members.AsQueryable(); // direct parameter usage
    }
}
```

**src-managedcode-dotnet-skills** — **All three apps** use primary constructors:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/FitnessStudioApi/Services/MemberService.cs
public sealed class MemberService(FitnessDbContext db, ILogger<MemberService> logger) : IMemberService
{
    public async Task<PagedResult<MemberResponse>> GetAllAsync(
        string? search, bool? isActive, int page, int pageSize, CancellationToken ct)
    {
        var query = db.Members.AsNoTracking().AsQueryable();
    }
}
```

### Verdict

**`src-dotnet-artisan` and `src-managedcode-dotnet-skills` are strongest** — both achieve 100% adoption of C# 12 primary constructors, eliminating boilerplate field declarations. `src-dotnet-webapi` is inconsistent (FitnessStudioApi uses traditional constructors). `src-no-skills` doesn't use the feature at all, producing noticeably more verbose code.

## 4. DTO Design — Records vs Classes, Naming Conventions

### What each directory does

**src-no-skills** — All DTOs are **mutable classes** with `{ get; set; }` properties. Uses `*Dto` naming:

```csharp
// src-no-skills/FitnessStudioApi/DTOs/MemberDtos.cs
public class MemberDto
{
    public int Id { get; set; }
    public string FirstName { get; set; } = string.Empty;
    public string LastName { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
}
```

**src-dotnet-webapi** — **`sealed record`** types with `*Request`/`*Response` naming. Responses use positional syntax; requests use `init` properties with validation attributes:

```csharp
// src-dotnet-webapi/FitnessStudioApi/DTOs/MemberDtos.cs
public sealed record MemberResponse(
    int Id, string FirstName, string LastName, string Email,
    string Phone, DateOnly DateOfBirth, DateOnly JoinDate,
    bool IsActive, DateTime CreatedAt, DateTime UpdatedAt);

public sealed record CreateMemberRequest
{
    [Required, MaxLength(100)]
    public required string FirstName { get; init; }
    [Required, EmailAddress]
    public required string Email { get; init; }
}
```

**src-dotnet-artisan** — **`record`** types (not sealed) with `*Request`/`*Response` naming:

```csharp
// src-dotnet-artisan/FitnessStudioApi/DTOs/Dtos.cs
public record CreateMembershipPlanRequest(
    string Name, string? Description, int DurationMonths,
    decimal Price, int MaxClassBookingsPerWeek, bool AllowsPremiumClasses);

public record MembershipPlanResponse(
    int Id, string Name, string? Description, int DurationMonths,
    decimal Price, int MaxClassBookingsPerWeek, bool AllowsPremiumClasses,
    bool IsActive, DateTime CreatedAt, DateTime UpdatedAt);
```

VetClinicApi DTOs are `sealed record`:

```csharp
// src-dotnet-artisan/VetClinicApi/DTOs/OwnerDtos.cs
public sealed record OwnerResponse(int Id, string FirstName, string LastName, ...);
public sealed record CreateOwnerRequest { ... }
```

**src-managedcode-dotnet-skills** — FitnessStudioApi and VetClinicApi use **`sealed record`** with `*Request`/`*Response`; LibraryApi uses plain **`record`** with `*Dto` naming:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/FitnessStudioApi/DTOs/MemberDtos.cs
public sealed record MemberResponse(int Id, string FirstName, string LastName, ...);
public sealed class CreateMemberRequest { [Required] public string FirstName { get; init; } ... }

// src-managedcode-dotnet-skills/LibraryApi/DTOs/BookDtos.cs
public record BookDto(int Id, string Title, string ISBN, ...);
public record CreateBookDto { ... }
```

### Verdict

**`src-dotnet-webapi` is the strongest** — it consistently uses `sealed record` (immutable by design, value-based equality), separates positional responses from `init`-property requests, and uses the clearer `*Request`/`*Response` convention throughout. Records are idiomatic for DTOs in modern C#. `src-no-skills`'s mutable class DTOs are the weakest — they invite accidental mutation and miss out on structural equality.

## 5. Service Abstraction — Interface + Implementation vs Concrete-Only

### What each directory does

All four directories use the **interface + implementation** pattern consistently:

```csharp
// All directories register services the same way
builder.Services.AddScoped<IMemberService, MemberService>();
builder.Services.AddScoped<IBookingService, BookingService>();
```

The differences are in **file organization**:

| Directory | Interface File Pattern |
|---|---|
| `src-no-skills` | Separate `Interfaces/I*Service.cs` folder (Fitness), monolithic `Interfaces.cs` (Library), individual `I*Service.cs` files (VetClinic) |
| `src-dotnet-webapi` | Individual `I*Service.cs` files per entity |
| `src-dotnet-artisan` | Interface + implementation **in the same file** (Fitness/Library), separate files (VetClinic) |
| `src-managedcode-dotnet-skills` | Separate `Interfaces/` subfolder (Fitness), individual files (Library/VetClinic) |

### Verdict

**All directories are equivalent** on the pattern itself. For file organization, individual `I*Service.cs` files (used by `src-dotnet-webapi` and `src-dotnet-artisan`/VetClinic) offer the best discoverability. Co-locating interface + implementation in one file (`src-dotnet-artisan`/Fitness) reduces file count but makes interfaces harder to find.

## 6. CancellationToken Propagation

### What each directory does

**src-no-skills** — **Zero** `CancellationToken` usage anywhere. Not in controllers, not in services, not in EF Core calls.

**src-dotnet-webapi** — **Comprehensive**: CancellationToken flows from endpoint → service interface → service implementation → EF Core calls across all three apps:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
group.MapGet("/", async Task<Ok<PaginatedResponse<MemberResponse>>> (
    IMemberService service, string? search = null,
    int page = 1, int pageSize = 20,
    CancellationToken ct = default) =>
{
    var result = await service.GetAllAsync(search, isActive, page, pageSize, ct);
    return TypedResults.Ok(result);
})

// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
public async Task<PaginatedResponse<MemberResponse>> GetAllAsync(
    string? search, bool? isActive, int page, int pageSize, CancellationToken ct)
{
    var query = _db.Members.AsNoTracking().AsQueryable();
    var totalCount = await query.CountAsync(ct);
    var items = await query.Skip(...).Take(...).ToListAsync(ct);
}
```

**src-dotnet-artisan** — **Only VetClinicApi** propagates CancellationToken; FitnessStudioApi and LibraryApi omit it entirely:

```csharp
// src-dotnet-artisan/VetClinicApi/Controllers/OwnersController.cs
public async Task<ActionResult<PagedResult<OwnerResponse>>> GetAll(
    [FromQuery] string? search, [FromQuery] int page = 1,
    [FromQuery] int pageSize = 10, CancellationToken ct = default)
{
    var result = await ownerService.GetAllAsync(search, page, pageSize, ct);
    return Ok(result);
}

// src-dotnet-artisan/FitnessStudioApi/Controllers/MembersController.cs (NO ct)
public async Task<IActionResult> GetAll([FromQuery] string? search,
    [FromQuery] bool? isActive, [FromQuery] PaginationParams pagination)
    => Ok(await service.GetAllAsync(search, isActive, pagination));
```

**src-managedcode-dotnet-skills** — FitnessStudioApi and VetClinicApi propagate CancellationToken; LibraryApi does not:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/.../Endpoints/MemberEndpoints.cs
group.MapGet("/", async (string? search, bool? isActive, int page, int pageSize,
    IMemberService service, CancellationToken ct) => { ... })

// src-managedcode-dotnet-skills/LibraryApi/Controllers/BooksController.cs (NO ct)
public async Task<ActionResult<PagedResult<BookDto>>> GetAll(
    [FromQuery] int page = 1, [FromQuery] int pageSize = 10)
{
    var result = await bookService.GetAllAsync(page, pageSize);
    return Ok(result);
}
```

### Verdict

**`src-dotnet-webapi` is the clear winner** — 100% CancellationToken propagation across all apps and layers. This is critical for production APIs: without token forwarding, cancelled HTTP requests continue executing database queries and business logic, wasting server resources. The inconsistency in `src-dotnet-artisan` and `src-managedcode-dotnet-skills` is a red flag — it means some apps handle cancellation while others don't.

## 7. AsNoTracking() in Read Queries

### What each directory does

**src-no-skills** — **Never** uses `AsNoTracking()`. All queries include change tracking overhead:

```csharp
// src-no-skills/FitnessStudioApi/Services/MemberService.cs
var query = _context.Members.AsQueryable(); // change tracking enabled
```

**src-dotnet-webapi** — **Consistent** `AsNoTracking()` on every read query across all three apps:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
var query = _db.Members.AsNoTracking().AsQueryable();
```

**src-dotnet-artisan** — Only VetClinicApi uses `AsNoTracking()`; FitnessStudioApi and LibraryApi do not:

```csharp
// src-dotnet-artisan/VetClinicApi/Services/OwnerService.cs
var query = db.Owners.AsNoTracking();

// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs (NO AsNoTracking)
var query = db.Members.AsQueryable();
```

**src-managedcode-dotnet-skills** — **Consistent** `AsNoTracking()` across all three apps:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/.../Services/MemberService.cs
var query = db.Members.AsNoTracking().AsQueryable();

// src-managedcode-dotnet-skills/LibraryApi/Services/BookService.cs
var items = await context.Books.AsNoTracking()
    .Include(b => b.BookAuthors).ThenInclude(ba => ba.Author)
    .Include(b => b.BookCategories).ThenInclude(bc => bc.Category)
    ...
```

### Verdict

**`src-dotnet-webapi` and `src-managedcode-dotnet-skills` tie** — both consistently apply `AsNoTracking()` to read queries. EF Core's change tracker allocates memory for every tracked entity; for read-only API responses this is pure overhead. Microsoft's documentation explicitly recommends `AsNoTracking()` for read-only scenarios. The `dotnet-entity-framework-core` skill in the managedcode set likely drives this behavior.

## 8. Return Type Precision

### What each directory does

**src-no-skills** — Uses **`List<T>`** everywhere:

```csharp
// src-no-skills/FitnessStudioApi/DTOs/PaginationDtos.cs
public class PagedResult<T>
{
    public List<T> Items { get; set; } = new();
}
```

**src-dotnet-webapi** — Uses **`IReadOnlyList<T>`** in paginated responses and service return types:

```csharp
// src-dotnet-webapi/FitnessStudioApi/DTOs/PaginatedResponse.cs
public sealed record PaginatedResponse<T>(
    IReadOnlyList<T> Items, int Page, int PageSize,
    int TotalCount, int TotalPages);

// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
public async Task<IReadOnlyList<BookingResponse>> GetUpcomingBookingsAsync(
    int memberId, CancellationToken ct) { ... }
```

**src-dotnet-artisan** — **Mixed**: VetClinicApi and FitnessStudioApi use `IReadOnlyList<T>` in pagination/DTOs; LibraryApi uses it in DTOs but `List<T>` in service method signatures:

```csharp
// src-dotnet-artisan/VetClinicApi/DTOs/PagedResult.cs
public sealed record PagedResult<T>(IReadOnlyList<T> Items, ...);

// src-dotnet-artisan/LibraryApi/Services/BookService.cs
Task<List<LoanResponse>> GetBookLoansAsync(int bookId); // List<T>, not IReadOnlyList<T>
```

**src-managedcode-dotnet-skills** — **Consistent** `IReadOnlyList<T>`:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/.../DTOs/PaginationDtos.cs
public sealed record PagedResult<T>(IReadOnlyList<T> Items, ...);

// src-managedcode-dotnet-skills/VetClinicApi/Services/IPetService.cs
Task<IReadOnlyList<MedicalRecordResponse>> GetMedicalRecordsAsync(int petId, CancellationToken ct);
Task<IReadOnlyList<VaccinationResponse>> GetVaccinationsAsync(int petId, CancellationToken ct);
```

### Verdict

**`src-dotnet-webapi` and `src-managedcode-dotnet-skills` are strongest** — `IReadOnlyList<T>` communicates intent ("this collection is not meant to be modified") and prevents consumers from accidentally adding or removing items. This follows the principle of least privilege for return types.

## 9. Data Seeder Design

### What each directory does

All configurations primarily use **static methods**, but with notable differences:

| Directory | Pattern | Example |
|---|---|---|
| `src-no-skills` | Static `async Task SeedAsync(DbContext)` | `await DataSeeder.SeedAsync(db);` |
| `src-dotnet-webapi` | Mixed — FitnessStudioApi: no seeder (model defaults only); LibraryApi/VetClinicApi: `HasData()` in `OnModelCreating` | `modelBuilder.Entity<Author>().HasData(...)` |
| `src-dotnet-artisan` | Static methods (sync and async variants) | `DataSeeder.Seed(db);` / `await SeedData.InitializeAsync(db);` |
| `src-managedcode-dotnet-skills` | Static `async Task InitializeAsync(DbContext)` | `await SeedData.InitializeAsync(db);` |

```csharp
// src-dotnet-webapi/LibraryApi/Data/LibraryDbContext.cs — HasData pattern
private static void SeedData(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Author>().HasData(
        new Author { Id = 1, FirstName = "Jane", LastName = "Austen", ... });
}

// src-managedcode-dotnet-skills/FitnessStudioApi/.../Data/SeedData.cs — runtime seeder
public static class SeedData
{
    public static async Task InitializeAsync(FitnessDbContext db)
    {
        if (await db.MembershipPlans.AnyAsync()) return;
        db.MembershipPlans.AddRange(CreatePlans());
        await db.SaveChangesAsync();
    }
}
```

### Verdict

**`src-dotnet-webapi`'s `HasData()` pattern** is the strongest for seed data that should be part of the schema (it integrates with EF migrations), while the runtime static seeder approach used by others is more flexible for development/demo scenarios. Both are valid depending on context.

## 10. Middleware Style — Convention-Based vs `IExceptionHandler`

### What each directory does

**src-no-skills** — **Convention-based middleware** with `RequestDelegate` and `InvokeAsync`:

```csharp
// src-no-skills/FitnessStudioApi/Middleware/GlobalExceptionHandlerMiddleware.cs
public class GlobalExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    public async Task InvokeAsync(HttpContext context)
    {
        try { await _next(context); }
        catch (Exception ex) { await HandleExceptionAsync(context, ex); }
    }
}
```

**src-dotnet-webapi** — **`IExceptionHandler`** (modern .NET 8+ pattern) in all three apps:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Middleware/ApiExceptionHandler.cs
public sealed class ApiExceptionHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken cancellationToken)
    {
        var (statusCode, title) = exception switch
        {
            BusinessRuleException bre => (bre.StatusCode, bre.Message),
            KeyNotFoundException => (StatusCodes.Status404NotFound, exception.Message),
            _ => (StatusCodes.Status500InternalServerError, "An unexpected error occurred")
        };
        ...
    }
}
```

**src-dotnet-artisan** — **Mixed**: FitnessStudioApi and VetClinicApi use `IExceptionHandler`; LibraryApi uses convention-based middleware:

```csharp
// src-dotnet-artisan/FitnessStudioApi/Middleware/GlobalExceptionHandler.cs
public class GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger) : IExceptionHandler { ... }

// src-dotnet-artisan/LibraryApi/Middleware/ExceptionHandlingMiddleware.cs
public class ExceptionHandlingMiddleware(RequestDelegate next, ILogger<...> logger)
{
    public async Task InvokeAsync(HttpContext context) { ... }
}
```

**src-managedcode-dotnet-skills** — FitnessStudioApi uses `IExceptionHandler`; LibraryApi and VetClinicApi use **inline exception handler** in `Program.cs`:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/.../Middleware/GlobalExceptionHandler.cs
public sealed class GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(...)
    {
        var (statusCode, title) = exception switch
        {
            BusinessRuleException => (StatusCodes.Status400BadRequest, "Business Rule Violation"),
            KeyNotFoundException => (StatusCodes.Status404NotFound, "Resource Not Found"),
            _ => (StatusCodes.Status500InternalServerError, "Internal Server Error")
        };
    }
}

// src-managedcode-dotnet-skills/LibraryApi/Program.cs (inline handler)
app.UseExceptionHandler(exceptionApp =>
{
    exceptionApp.Run(async context =>
    {
        var exception = context.Features.Get<IExceptionHandlerFeature>()?.Error;
        var (statusCode, title) = exception switch { ... };
    });
});
```

### Verdict

**`src-dotnet-webapi` is the strongest** — `IExceptionHandler` is the idiomatic .NET 8+ approach. It integrates with the framework's error handling pipeline, supports DI natively, and is chainable (multiple handlers can be registered). Convention-based middleware (`src-no-skills`) works but bypasses the framework's built-in error handling infrastructure. The inline lambda approach in `src-managedcode-dotnet-skills` is functional but harder to test.

## 11. Exception Handling Strategy

### What each directory does

**src-no-skills** — Uses **built-in exceptions** (`KeyNotFoundException`, `InvalidOperationException`) with pattern matching in middleware. LibraryApi additionally uses a **tuple result pattern**:

```csharp
// src-no-skills/LibraryApi/Services/BookService.cs
public async Task<(bool Success, string? Error)> DeleteBookAsync(int id)
{
    var book = await _db.Books.FindAsync(id);
    if (book == null) return (false, "Book not found.");
    if (hasActiveLoans) return (false, "Cannot delete book with active loans.");
}
```

**src-dotnet-webapi** — **Custom `BusinessRuleException`** with configurable status codes:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Middleware/ApiExceptionHandler.cs
public sealed class BusinessRuleException : Exception
{
    public int StatusCode { get; }
    public BusinessRuleException(string message, int statusCode = StatusCodes.Status400BadRequest)
        : base(message) { StatusCode = statusCode; }
}

// Usage in services:
throw new BusinessRuleException("Member must be at least 16 years old.");
throw new BusinessRuleException($"Email '{request.Email}' already exists.", 409);
```

**src-dotnet-artisan** — FitnessStudioApi uses **custom exceptions** (`BusinessRuleException`, `NotFoundException`); LibraryApi and VetClinicApi use built-in exceptions:

```csharp
// src-dotnet-artisan/FitnessStudioApi/Middleware/GlobalExceptionHandler.cs
public class BusinessRuleException(string message) : Exception(message);
public class NotFoundException(string message) : Exception(message);

// src-dotnet-artisan/VetClinicApi (built-in only — pattern-matched in middleware)
var (statusCode, title) = exception switch
{
    InvalidOperationException => (StatusCodes.Status400BadRequest, "Bad Request"),
    KeyNotFoundException => (StatusCodes.Status404NotFound, "Not Found"),
    _ => (StatusCodes.Status500InternalServerError, "Internal Server Error"),
};
```

**src-managedcode-dotnet-skills** — FitnessStudioApi uses custom `BusinessRuleException`; others use built-in exceptions:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/.../Services/BusinessRuleException.cs
public sealed class BusinessRuleException(string message) : Exception(message);
```

### Verdict

**`src-dotnet-webapi`'s `BusinessRuleException` with configurable status codes** is the most sophisticated — it allows services to communicate HTTP semantics without coupling to `Microsoft.AspNetCore.Http`. The simpler built-in exception approach (used by VetClinicApi in `src-dotnet-artisan`) is also valid but loses granularity (e.g., `InvalidOperationException` maps to 400 for everything). The tuple pattern in `src-no-skills`/LibraryApi adds complexity to controllers without offering clear advantages.

## 13. Pagination Implementation

### What each directory does

**src-no-skills** — Mutable **class** with `List<T>` items:

```csharp
// src-no-skills/FitnessStudioApi/DTOs/PaginationDtos.cs
public class PagedResult<T>
{
    public List<T> Items { get; set; } = new();
    public int TotalCount { get; set; }
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasPrevious => Page > 1;
    public bool HasNext => Page < TotalPages;
}
```

**src-dotnet-webapi** — **`sealed record`** with `IReadOnlyList<T>`, no computed navigation properties:

```csharp
// src-dotnet-webapi/FitnessStudioApi/DTOs/PaginatedResponse.cs
public sealed record PaginatedResponse<T>(
    IReadOnlyList<T> Items, int Page, int PageSize,
    int TotalCount, int TotalPages);
```

**src-dotnet-artisan** — Mixed: FitnessStudioApi uses a mutable **class** with computed properties; VetClinicApi uses a **`sealed record`**:

```csharp
// src-dotnet-artisan/FitnessStudioApi/DTOs/Dtos.cs
public class PaginatedResponse<T>
{
    public IReadOnlyList<T> Items { get; init; } = [];
    public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
    public bool HasPreviousPage => Page > 1;
    public bool HasNextPage => Page < TotalPages;
}

// src-dotnet-artisan/VetClinicApi/DTOs/PagedResult.cs
public sealed record PagedResult<T>(IReadOnlyList<T> Items, int TotalCount, int Page, int PageSize)
{
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasPrevious => Page > 1;
    public bool HasNext => Page < TotalPages;
}
```

**src-managedcode-dotnet-skills** — **`sealed record`** with computed navigation properties:

```csharp
// src-managedcode-dotnet-skills/FitnessStudioApi/.../DTOs/PaginationDtos.cs
public sealed record PagedResult<T>(IReadOnlyList<T> Items, int TotalCount, int Page, int PageSize)
{
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasPrevious => Page > 1;
    public bool HasNext => Page < TotalPages;
}
```

### Verdict

**`src-managedcode-dotnet-skills` is the strongest** — `sealed record` for immutability, `IReadOnlyList<T>` items, and computed `HasPrevious`/`HasNext` navigation properties. `src-dotnet-webapi`'s approach is good but precomputes `TotalPages` at the service level rather than deriving it, creating a slight data integrity risk.

## 14. OpenAPI / Swagger Metadata

### What each directory does

**src-no-skills** — **Rich**: Uses XML doc comments (`/// <summary>`) and `[ProducesResponseType]` attributes:

```csharp
// src-no-skills/FitnessStudioApi/Controllers/MembersController.cs
/// <summary>List members with search, filter, and pagination</summary>
[HttpGet]
[ProducesResponseType(typeof(PagedResult<MemberDto>), 200)]
public async Task<IActionResult> GetAll(...) { ... }
```

**src-dotnet-webapi** — **Richest**: Uses `.WithName()`, `.WithSummary()`, `.WithDescription()`, `.Produces<T>()`:

```csharp
// src-dotnet-webapi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
.WithName("GetMembers")
.WithSummary("List members")
.WithDescription("Returns a paginated list of members with optional search and filter.")
.Produces<PaginatedResponse<MemberResponse>>(200);

.WithName("GetMember")
.Produces<MemberDetailResponse>(200)
.Produces(404);
```

**src-dotnet-artisan** — **Mixed**: FitnessStudioApi has rich `[ProducesResponseType]` attributes; LibraryApi and VetClinicApi rely on generic `ActionResult<T>` only:

```csharp
// src-dotnet-artisan/FitnessStudioApi/Controllers/MembersController.cs (RICH)
[ProducesResponseType<PaginatedResponse<MemberResponse>>(200)]

// src-dotnet-artisan/LibraryApi/Controllers/AuthorsController.cs (MINIMAL)

```

**src-managedcode-dotnet-skills** — **Moderate**: Minimal APIs use `.WithName()` and `.WithSummary()` but no `.Produces<T>()` or `.WithDescription()`:

```csharp
// src-managedcode-dotnet-skills/VetClinicApi/Endpoints/PetEndpoints.cs
group.MapGet("/", GetAll).WithName("GetPets").WithSummary("Get all pets with pagination");
group.MapPost("/", Create).WithName("CreatePet").WithSummary("Create a new pet");
```

### Verdict

**`src-dotnet-webapi` is the strongest** — it combines `WithName()` (for stable operation IDs), `WithSummary()` and `WithDescription()` (for human-readable docs), and `.Produces<T>(statusCode)` (for accurate response schemas). This produces the highest-quality OpenAPI documents. `src-managedcode-dotnet-skills` lacks the `Produces<T>()` declarations, which means response schemas must be inferred by the framework.

## Summary: Impact of Skills

### Ranking by Overall Code Quality

| Rank | Directory | Score | Key Strengths |
|---|---|---|---|
| **1** | `src-dotnet-webapi` | ⭐⭐⭐⭐⭐ | Consistent Minimal APIs, `sealed` everywhere, full CancellationToken propagation, `AsNoTracking()` on all reads, `IExceptionHandler`, rich OpenAPI, `sealed record` DTOs, custom `BusinessRuleException` with status codes |
| **2** | `src-managedcode-dotnet-skills` | ⭐⭐⭐⭐ | Primary constructors everywhere, `AsNoTracking()` on all reads, `sealed` on most apps, `IExceptionHandler` (partial), CancellationToken (partial), `sealed record` DTOs, modern `[]` syntax |
| **3** | `src-dotnet-artisan` | ⭐⭐⭐ | Primary constructors everywhere, modern `[]` syntax, VetClinicApi is excellent (sealed, CancellationToken, AsNoTracking, IExceptionHandler), but other apps miss many best practices |
| **4** | `src-no-skills` | ⭐⭐ | Solid fundamentals (interfaces, async/await, ProblemDetails) but misses nearly every modern C# and ASP.NET Core best practice |

### Most Impactful Differences

1. **CancellationToken propagation** — The single most impactful production-readiness signal. Without it, cancelled requests waste server resources. `src-dotnet-webapi` is the only configuration achieving 100% coverage.

2. **`AsNoTracking()` on read queries** — Direct performance impact under load. EF Core's change tracker allocates per-entity state; omitting `AsNoTracking()` on read-only queries can double memory usage for large result sets.

3. **`sealed` types** — Enables JIT devirtualization optimizations. Consistently applied by `src-dotnet-webapi` only.

4. **`IExceptionHandler` vs convention-based middleware** — `IExceptionHandler` integrates with the ASP.NET Core error pipeline, supports DI, and is composable. Convention-based middleware bypasses this infrastructure.

5. **Record DTOs vs class DTOs** — Records provide immutability, value-based equality, and concise syntax. `src-no-skills`'s mutable class DTOs are the most significant design gap.

### Overall Assessment

The **`dotnet-webapi` skill** produces the most consistently high-quality code across all dimensions. It applies modern best practices uniformly to all three apps, making it the strongest single-skill configuration.

The **`managedcode-dotnet-skills` set** is the second strongest, particularly excelling in EF Core usage (`AsNoTracking()` everywhere) and modern C# features. Its main weakness is inconsistency — the LibraryApi app misses several patterns (no `sealed`, no CancellationToken, no `IExceptionHandler`).

The **`dotnet-artisan` skills** show a clear inconsistency pattern — VetClinicApi is excellent (matching `src-dotnet-webapi` quality), but FitnessStudioApi and LibraryApi miss critical patterns like `AsNoTracking()`, CancellationToken, and `sealed`. This suggests the skills provide guidance that the model applies unevenly.

The **no-skills baseline** produces functional, well-structured code but consistently defaults to older patterns — classes instead of records, no `sealed`, no `AsNoTracking()`, no CancellationToken, convention-based middleware, and `new List<T>()` instead of `[]`. It demonstrates that without skill guidance, the model falls back to pre-C# 12 / pre-.NET 8 patterns.
