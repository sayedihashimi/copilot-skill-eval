# Comparative Analysis: no-skills, dotnet-artisan, managedcode-dotnet-skills, dotnet-skills, dotnet-webapi

## Introduction

This report compares the **VetClinicApi** (a veterinary clinic management API) generated by **five different Copilot skill configurations** in run-3:

| Configuration | Directory | Description |
|---|---|---|
| **no-skills** | `output/no-skills/run-3/` | Baseline — default Copilot with no custom skills |
| **dotnet-artisan** | `output/dotnet-artisan/run-3/` | dotnet-artisan plugin chain (using-dotnet, dotnet-advisor, dotnet-csharp, dotnet-api) |
| **managedcode-dotnet-skills** | `output/managedcode-dotnet-skills/run-3/` | Community managed-code skills (dotnet-aspnet-core, dotnet-entity-framework-core, dotnet-modern-csharp, dotnet-project-setup) |
| **dotnet-skills** | `output/dotnet-skills/run-3/` | Official .NET Skills (optimizing-ef-core-queries, analyzing-dotnet-performance) |
| **dotnet-webapi** | `output/dotnet-webapi/run-3/` | dotnet-webapi skill (purpose-built for ASP.NET Core Web API generation) |

Each configuration generated the same VetClinicApi scenario: a booking/scheduling system for "Happy Paws Veterinary Clinic" with 7 entities, 35 endpoints, and complex business rules including appointment conflict detection, status workflow state machines, soft-delete, and vaccination tracking.

All configurations use guard clauses in services, but none use the modern `ArgumentNullException.ThrowIfNull()` pattern:

## 6. EF Migration Usage [CRITICAL]

This is one of the most critical dimensions — `EnsureCreated()` prevents schema evolution and is an anti-pattern for anything beyond prototyping.

**no-skills / dotnet-artisan / managedcode / dotnet-skills** — All use EnsureCreated:
```csharp
// Common anti-pattern in 4 configs
await db.Database.EnsureCreatedAsync();
await DataSeeder.SeedAsync(db);
```

**dotnet-webapi** — Uses proper EF Core migrations:
```csharp
// dotnet-webapi: Program.cs
using (var scope = app.Services.CreateScope())
{
    var dbContext = scope.ServiceProvider.GetRequiredService<VetClinicDbContext>();
    dbContext.Database.Migrate();
}
```
With actual migration files in `Migrations/`:
```
Migrations/
├── 20260329190318_InitialCreate.cs
├── 20260329190318_InitialCreate.Designer.cs
└── VetClinicDbContextModelSnapshot.cs
```

| Config | Score | Justification |
|---|---|---|
| no-skills | 1 | Uses EnsureCreated — no migration support |
| dotnet-artisan | 1 | Uses EnsureCreatedAsync — no migration support |
| managedcode | 1 | Uses EnsureCreatedAsync — no migration support |
| dotnet-skills | 1 | Uses EnsureCreatedAsync — no migration support |
| dotnet-webapi | 5 | Uses Database.Migrate() with actual migration files and HasData() seeding |

**Verdict:** **dotnet-webapi** is the only configuration that correctly uses EF Core migrations. This is a binary distinction — you either use migrations or you don't — and only dotnet-webapi gets it right. The migration file includes all 39 seed records via `HasData()`, making the schema reproducible.

public class GlobalExceptionHandler : IExceptionHandler

            ArgumentException => (StatusCodes.Status400BadRequest, "Bad Request"),
            InvalidOperationException => (StatusCodes.Status409Conflict, "Conflict"),
            _ => (0, (string?)null)
        };
    }
}

### Most Impactful Differences

1. **EF Migrations vs EnsureCreated** (30-point swing): Only dotnet-webapi uses migrations — this alone accounts for the largest scoring gap. All others use the `EnsureCreated()` anti-pattern that makes schema evolution impossible.

2. **Minimal API vs Controllers** (36-point swing across CRITICAL dimensions): dotnet-webapi and dotnet-artisan use Minimal APIs; the other three use controllers. This affects API architecture, documentation, and return type safety.

3. **CancellationToken propagation** (6-point swing per config): Only dotnet-artisan and dotnet-webapi thread CancellationToken through the entire call chain. The others waste server resources on cancelled HTTP requests.

4. **Modern C# patterns** (6-point swing per config): Primary constructors, sealed records, collection expressions — dotnet-artisan and dotnet-webapi apply these consistently. no-skills and dotnet-skills use traditional C# patterns.

5. **Sealed types** (5-point swing): dotnet-artisan and dotnet-webapi seal all types for JIT optimization. managedcode and no-skills seal nothing.

### Overall Assessment

**dotnet-webapi** delivers the most production-ready output by a significant margin. It is the only configuration that uses EF Core migrations, `Results<T1, T2>` union types for automatic OpenAPI schema generation, `HasData()` for migration-integrated seeding, Scalar instead of Swashbuckle, and `internal sealed` on infrastructure types. It represents the current gold standard for .NET Web API generation.

**dotnet-artisan** is the strongest runner-up, matching dotnet-webapi on modern C# adoption, sealed types, and CancellationToken propagation. Its main gap is the use of `EnsureCreated()` instead of migrations.

**managedcode-dotnet-skills** occupies a middle ground — it uniquely implements a global query filter for soft-delete and is the only config with HTTPS redirection, but falls short on API architecture (controllers) and CancellationToken.

**dotnet-skills** adds value through FluentValidation and advanced EF Core query patterns (AsSplitQuery, EF.Functions.Like) but uses traditional C# patterns and controllers.

**no-skills** (baseline) produces functional code but misses every modern .NET practice — no Minimal APIs, no sealed types, no primary constructors, no CancellationToken, no AsNoTracking, and no migrations. It demonstrates the gap that skills are designed to close.
