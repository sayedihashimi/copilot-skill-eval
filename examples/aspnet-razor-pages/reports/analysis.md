# Comparative Analysis: no-skills, dotnet-webapi, dotnet-artisan, managedcode-dotnet-skills

## Introduction

This report compares four ASP.NET Core Razor Pages application sets, each generated by Copilot under a different skill configuration. All four configurations produced the same three applications—**SparkEvents** (event registration portal), **KeystoneProperties** (property management), and **HorizonHR** (employee directory & HR portal)—targeting .NET 10 with EF Core + SQLite and Bootstrap 5.

| Configuration | Directory | Skills Used |
|---|---|---|
| **no-skills** | `output/no-skills/` | All 11 skills combined (dotnet-artisan plugin chain + dotnet-webapi + managedcode-dotnet-skills) |
| **dotnet-webapi** | `output/dotnet-webapi/` | `dotnet-webapi` skill only |
| **dotnet-artisan** | `output/dotnet-artisan/` | dotnet-artisan plugin chain (5 skills: using-dotnet → dotnet-advisor → dotnet-csharp + dotnet-api + dotnet-ui) |
| **managedcode-dotnet-skills** | `output/managedcode-dotnet-skills/` | Community managed-code skills |

Each app was built by an isolated agent with no shared context. All 12 resulting projects compile with 0 errors and 0 warnings.

## 1. Page Model Design

Page models are the core unit of a Razor Pages app. Thin models that delegate to services are more testable and maintainable.

### no-skills — Thin, sealed, primary constructors

```csharp
// no-skills/SparkEvents/Pages/Events/Index.cshtml.cs
public sealed class IndexModel(
    IEventService eventService,
    ICategoryService categoryService) : PageModel
{
    public async Task OnGetAsync(CancellationToken ct)
    {
        Events = await eventService.GetAllAsync(
            SearchTerm, CategoryId, Status, StartDate, EndDate, PageNumber, 10, ct);
    }
}
```

### dotnet-webapi — Thin, sealed, traditional constructors

```csharp
// dotnet-webapi/SparkEvents/Pages/Index.cshtml.cs
public sealed class IndexModel : PageModel
{
    private readonly IEventService _eventService;
    private readonly IRegistrationService _registrationService;
    private readonly IAttendeeService _attendeeService;

    public IndexModel(IEventService eventService,
        IRegistrationService registrationService,
        IAttendeeService attendeeService)
    {
        _eventService = eventService;
        _registrationService = registrationService;
        _attendeeService = attendeeService;
    }
}
```

### dotnet-artisan — Fat models with DbContext leak

```csharp
// dotnet-artisan/SparkEvents/Pages/Events/Index.cshtml.cs
public class IndexModel(IEventService eventService, SparkEventsDbContext db) : PageModel
{
    public async Task OnGetAsync(...)
    {
        // Direct DbContext queries in the page model — violates separation of concerns
        Categories = await db.EventCategories.OrderBy(c => c.Name).ToListAsync();
        Events = await eventService.GetEventsAsync(...);
    }
}
```

### managedcode-dotnet-skills — Thin, primary constructors

```csharp
// managedcode-dotnet-skills/SparkEvents/Pages/Events/Index.cshtml.cs
public class IndexModel(IEventService eventService, ICategoryService categoryService) : PageModel
{
    public async Task OnGetAsync(...)
    {
        Categories = await categoryService.GetAllCategoriesAsync();
        Events = await eventService.GetAllEventsAsync(search, status, categoryId, page, pageSize);
    }
}
```

**Verdict:** **no-skills** wins — thin models with sealed classes, primary constructors, and proper service delegation. dotnet-artisan's DbContext injection into page models is an anti-pattern that couples the presentation layer to the data layer and undermines testability.

## 2. Form Handling & Validation

Proper form handling prevents over-posting, CSRF attacks, and duplicate submissions.

All four configurations rely on the Razor Pages framework's **automatic antiforgery validation** for POST handlers (no explicit `[ValidateAntiForgeryToken]` attributes needed). All consistently follow the **PRG (Post-Redirect-Get)** pattern:

```csharp
// Consistent across ALL configs — PRG pattern
public async Task<IActionResult> OnPostAsync(...)
{
    if (!ModelState.IsValid)
    {
        await LoadSelectListsAsync();
        return Page();                          // ← stay on page for validation errors
    }

    // ... business logic ...

    TempData["SuccessMessage"] = "Event created.";
    return RedirectToPage("Details", new { id = evt.Id });  // ← redirect on success
}
```

### Validation summary differences

| Config | Pattern | Example |
|---|---|---|
| no-skills | `ModelOnly` consistently | `<div asp-validation-summary="ModelOnly" class="text-danger">` |
| dotnet-webapi | Mixed (`All` and `ModelOnly`) | Varies by app |
| dotnet-artisan | Mixed | Varies by app |
| managedcode-dotnet-skills | Mixed | Varies by app |

**Verdict:** All configurations handle form submission well. **no-skills** is slightly more consistent with its `ModelOnly` validation summary (the recommended default, since field-level errors are shown via `asp-validation-for`).

## 3. Input Model Separation

Input model separation prevents over-posting attacks by binding only the properties you expose, not the entire entity.

### no-skills & dotnet-webapi — Sealed nested InputModel

```csharp
// no-skills/SparkEvents/Pages/Events/Create.cshtml.cs
[BindProperty]
public InputModel Input { get; set; } = new();

public sealed class InputModel
{
    [Required, MaxLength(300)]
    public string Title { get; set; } = string.Empty;

    [Required, Display(Name = "Category")]
    public int EventCategoryId { get; set; }

    [Required, Range(1, int.MaxValue)]
    public int TotalCapacity { get; set; } = 100;
}
```

### dotnet-artisan — Non-sealed nested InputModel

```csharp
// dotnet-artisan/SparkEvents/Pages/Events/Create.cshtml.cs
[BindProperty]
public InputModel Input { get; set; } = new();

public class InputModel  // Not sealed
{
    [Required, MaxLength(300)]
    public string Title { get; set; } = string.Empty;
}
```

### managedcode-dotnet-skills — Semantically named InputModel

```csharp
// managedcode-dotnet-skills/SparkEvents/Pages/Events/Create.cshtml.cs
[BindProperty]
public EventInputModel Input { get; set; } = new();

public class EventInputModel  // Semantic name instead of generic "InputModel"
{
    [Required, MaxLength(300)]
    public string Title { get; set; } = string.Empty;
}
```

**Verdict:** All configs correctly use dedicated input models — none bind directly to entities. **no-skills/dotnet-webapi** are best with sealed nested classes. **managedcode-dotnet-skills** adds useful semantic naming (`EventInputModel` vs generic `InputModel`).

## 4. Named Handler Methods

Named handlers produce cleaner routing and more testable page models when a page has multiple form actions.

All configs use named handlers where appropriate (approve/reject, publish/complete, create/toggle):

```csharp
// dotnet-artisan/HorizonHR/Pages/Leave/Review.cshtml.cs
public async Task<IActionResult> OnPostApproveAsync(int id, CancellationToken ct)
{
    var (success, error) = await _leaveService.ApproveRequestAsync(id, ReviewerId, ReviewNotes, ct);
    TempData[success ? "SuccessMessage" : "ErrorMessage"] = success ? "Leave request approved." : error;
    return RedirectToPage("Details", new { id });
}

public async Task<IActionResult> OnPostRejectAsync(int id, CancellationToken ct)
{
    var (success, error) = await _leaveService.RejectRequestAsync(id, ReviewerId, ReviewNotes, ct);
    TempData[success ? "SuccessMessage" : "ErrorMessage"] = success ? "Leave request rejected." : error;
    return RedirectToPage("Details", new { id });
}
```

```csharp
// dotnet-webapi/SparkEvents/Pages/Events/Details.cshtml.cs
public async Task<IActionResult> OnPostPublishAsync(int id) { ... }
```

**Verdict:** Tie — all four configurations correctly use named handlers for multi-action pages. Approximately 25% of pages across all configs use named handlers; the rest use a single `OnPostAsync` where only one action exists.

## 7. View Components & Reusable UI

Component composition reduces duplication and improves testability.

| Approach | no-skills | dotnet-webapi | dotnet-artisan | managedcode-dotnet-skills |
|---|---|---|---|---|
| ViewComponent classes | ❌ | ✅ `PaginationViewComponent` | ❌ | ✅ `StatusBadgeViewComponent` |
| Custom TagHelpers | ❌ | ✅ `StatusBadgeTagHelper` | ❌ | ❌ |
| Partial views (status badges) | ✅ | ✅ | ✅ | ✅ |
| Partial views (pagination) | ✅ | ✅ | ✅ | ✅ |

### dotnet-webapi — Custom TagHelper for status badges

```csharp
// dotnet-webapi/SparkEvents/TagHelpers/StatusBadgeTagHelper.cs
[HtmlTargetElement("status-badge")]
public sealed class StatusBadgeTagHelper : TagHelper
{
    [HtmlAttributeName("status")]
    public string Status { get; set; } = string.Empty;

    public override void Process(TagHelperContext context, TagHelperOutput output)
    {
        output.TagName = "span";
        var badgeClass = Status switch
        {
            nameof(EventStatus.Published) => "bg-success",
            nameof(EventStatus.Cancelled) => "bg-danger",
            _ => "bg-secondary"
        };
        output.Attributes.SetAttribute("class", $"badge {badgeClass}");
        output.Content.SetContent(Status);
    }
}
```

Usage in views:

```html
<!-- dotnet-webapi — clean tag helper invocation -->
<td><status-badge status="@evt.Status.ToString()"></status-badge></td>
```

### managedcode-dotnet-skills — ViewComponent for status badges

```csharp
// managedcode-dotnet-skills/SparkEvents/Pages/Shared/Components/StatusBadge/StatusBadgeViewComponent.cs
public class StatusBadgeViewComponent : ViewComponent
{
    public IViewComponentResult Invoke(string status)
    {
        var (cssClass, displayText) = status switch
        {
            "Draft" => ("bg-secondary", "Draft"),
            "Published" => ("bg-success", "Published"),
            _ => ("bg-secondary", status)
        };
        ViewData["CssClass"] = cssClass;
        return View("Default");
    }
}
```

### no-skills & dotnet-artisan — Partial views only

```html
<!-- no-skills — status badge partial -->
@model SparkEvents.Models.EventStatus
@{
    var (badgeClass, label) = Model switch
    {
        EventStatus.Draft => ("bg-secondary", "Draft"),
        EventStatus.Published => ("bg-success", "Published"),
        EventStatus.Cancelled => ("bg-danger", "Cancelled"),
        _ => ("bg-secondary", Model.ToString())
    };
}
<span class="badge @badgeClass">@label</span>
```

**Verdict:** **dotnet-webapi** is the clear winner with both a custom TagHelper (`<status-badge>`) and a ViewComponent (`PaginationViewComponent`). TagHelpers are the Razor-idiomatic way to create reusable, encapsulated UI components that feel native in markup.

## 10. Tag Helper Usage

Tag helpers are the modern standard for Razor markup, producing cleaner templates than legacy `@Html.*` helpers.

All four configurations use modern `asp-*` tag helpers exclusively:

```html
<!-- Consistent across ALL configs -->
<input asp-for="Input.Title" class="form-control" />
<select asp-for="CategoryId" asp-items="Model.Categories" class="form-select">
    <option value="">All Categories</option>
</select>
<a asp-page="Details" asp-route-id="@evt.Id" class="btn btn-sm btn-outline-info">View</a>
<span asp-validation-for="Input.Title" class="text-danger"></span>
```

No legacy `@Html.TextBoxFor()`, `@Html.DropDownListFor()`, or `@Html.ActionLink()` patterns were found in any configuration. The only `@Html.*` usage is `Html.GetEnumSelectList<T>()` for enum dropdowns — which is the correct modern approach:

```html
<select asp-for="Status" asp-items="Html.GetEnumSelectList<EventStatus>()" class="form-select">
```

All `_ViewImports.cshtml` files register the built-in tag helpers:

```html
@addTagHelper *, Microsoft.AspNetCore.Mvc.TagHelpers
```

dotnet-webapi's SparkEvents additionally registers its custom tag helpers:

```html
@addTagHelper *, SparkEvents
```

**Verdict:** Tie — all configurations use modern tag helpers exclusively with no legacy patterns.

## 19. Data Seeder Design

Seed data strategies impact migration workflows, testability, and application startup.

| Aspect | no-skills | dotnet-webapi | dotnet-artisan | managedcode-dotnet-skills |
|---|---|---|---|---|
| Method signature | `async Task SeedAsync()` | `void Seed()` | `void Seed()` | `async Task SeedAsync()` |

| Class type | `static class` | `static class` | `static class` | `static class` |

### no-skills — Async, idempotent seeder

```csharp
// no-skills/SparkEvents/Data/DataSeeder.cs
public static class DataSeeder
{

        if (await db.EventCategories.AnyAsync()) return;  // Async idempotency

            new() { Name = "Technology", ColorHex = "#007BFF" },
            new() { Name = "Business", ColorHex = "#28A745" },
        };

    }
}
```

```csharp
// dotnet-webapi/KeystoneProperties/Data/DataSeeder.cs
public static class DataSeeder

        context.SaveChanges();  // Blocking I/O
    }
}
```

**Verdict:** **no-skills** and **managedcode-dotnet-skills** use async seeders, which avoid blocking the thread pool during startup. For production apps, async seeding scales better. All configs correctly implement idempotency guards.

### dotnet-webapi — Custom exception middleware with structured logging

```csharp

public sealed class GlobalExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<GlobalExceptionHandlerMiddleware> _logger;

    public async Task InvokeAsync(HttpContext context)
    {
        try
        {

        }
        catch (Exception ex)

    }

}
```

## 22. Pagination

```csharp
    public int TotalCount { get; }

    public bool HasNextPage => PageNumber < TotalPages;
```

### dotnet-artisan — Enhanced with static async factory

```csharp
// dotnet-artisan — CreateAsync helper method
public static async Task<PaginatedList<T>> CreateAsync(
    IQueryable<T> source, int pageNumber, int pageSize, CancellationToken ct = default)
{
    var count = await source.CountAsync(ct);
    var items = await source.Skip((pageNumber - 1) * pageSize)
        .Take(pageSize).ToListAsync(ct);
    return new PaginatedList<T>(items, count, pageNumber, pageSize);
}
```

### Most Impactful Differences (ranked by production importance)

1. **CancellationToken propagation** (§16) — no-skills and dotnet-artisan pass cancellation tokens through all layers; dotnet-webapi and managedcode omit them entirely. Critical for production scalability.

2. **AsNoTracking usage** (§17) — managedcode-dotnet-skills is the only config that consistently applies `AsNoTracking()` to read-only queries (69 occurrences). dotnet-artisan never uses it. This can halve memory consumption on list pages.

3. **Page model thickness** (§1) — dotnet-artisan injects `DbContext` directly into page models alongside services, creating fat models that violate separation of concerns. All other configs maintain thin page models.

4. **Sealed types** (§13) — no-skills and dotnet-webapi seal all 178–182 classes, enabling JIT devirtualization. dotnet-artisan and managedcode leave everything open.

5. **Reusable UI components** (§7, §24) — dotnet-webapi is the only config that creates both a custom TagHelper and a ViewComponent. Others rely solely on partial views.

6. **Accessibility** (§6) — dotnet-webapi leads with 184 ARIA markers. All configs miss `aria-describedby` and screen-reader-only text.

7. **File organization** (§21) — managedcode-dotnet-skills separates EF configurations into a `Configurations/` folder and enums into an `Enums/` folder — the most scalable approach.
